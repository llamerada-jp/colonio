== 概要

Colonio は WebRTC ベースの P2P オーバーレイネットワークを構築するライブラリです。
2つのコンポーネントから構成されます。

Seed(サーバー)::
ノードの登録管理と WebRTC シグナリングの中継を担当します。
ネットワークへの参加時に最初に接続するエントリーポイントです。

Node(クライアント)::
P2P ネットワークに参加し、他の Node と直接通信を行うクライアントです。
Seed へのブートストラップ完了後は、Node 間で WebRTC DataChannel による直接接続を確立し、パケットをリレーします。

Seed はノード管理とシグナリングのみを行い、ユーザーデータの中継には関与しません。
そのため、ネットワーク規模が大きくなっても Seed がボトルネックになりにくい設計です。

== システム構成

=== 接続フロー

Node がネットワークに参加する流れは以下のとおりです。

. Node が Seed に接続し、NodeID の割り当てを受ける
. Seed はネットワーク内の他の Node を紹介する
. Node 間で WebRTC シグナリング(Offer / Answer / ICE)を Seed 経由で交換する
. WebRTC DataChannel が確立され、以降は Node 間で直接通信する
. ルーティングテーブルを隣接 Node と交換し、オーバーレイネットワークのトポロジーを構築する

=== NodeID

各 Node には 128 ビットのランダムな NodeID が割り当てられます。
NodeID は Seed によって生成され、ネットワーク内で一意です。
NodeID の空間はリング状に扱われ、1D ルーティングの基盤となります。

=== Seed の内部構造

Seed は以下の3層で構成されます。

Server::
HTTP サーバー。Connect RPC(gRPC 互換)でリクエストを受け付けます。
セッション管理に Cookie を使用し、Node の認証を行います。

Controller::
ノードのライフサイクル管理を担当します。
ノードの割り当て・解放、Keepalive によるノードの生存確認、シグナリングメッセージのルーティングを行います。
各 Node には有効期限(Lifespan)が設定されており、Node は定期的に Keepalive を送信して期限を延長します。
Node がオフラインになり Keepalive が途絶えた場合、期限切れにより Seed が離脱を検知し、ネットワークから除去します。

Gateway::
ノードの永続化と参照を担当するデータアクセス層です。
デフォルトの実装(SimpleGateway)はインメモリで動作しますが、インタフェースを実装することで外部データストアに差し替えることができます。

=== Node の内部構造

Node は以下のモジュールで構成されます。

Network::
P2P 通信の基盤となるネットワーク層です。
Seed との通信、他 Node との WebRTC 接続、パケット転送、ルーティングを担当します。
詳細は <<ネットワーク層>> を参照してください。

Messaging::
NodeID を指定した 1 対 1 のメッセージ送受信を行うモジュールです。
詳細は <<機能: Messaging>> を参照してください。

Spread::
2D 座標上の指定範囲にメッセージをブロードキャストするモジュールです。
詳細は <<機能: Spread>> を参照してください。

== ライブラリとしての利用

Colonio は Seed・Node ともにスタンドアロンのプログラムではなく、ユーザーのアプリケーションに組み込んで使用するライブラリです。

=== Seed の組み込み

`seed.NewSeed()` でインスタンスを生成し、`RegisterService(mux)` で Go 標準の `http.ServeMux` に登録します。
HTTP サーバーの起動・TLS 設定・ポート管理はユーザーのアプリケーション側で行います。
これにより、既存のサーバーアプリケーションに Seed の機能を追加する形で利用できます。

Gateway インタフェースを差し替えることでノード管理のバックエンドをカスタマイズできます。

=== Node の組み込み

`colonio.NewColonio()` で Seed の URL や ICE サーバー等を設定してインスタンスを生成し、`Start()` でネットワークに参加します。
`Colonio` インタフェースを通じて Messaging・Spread 等の各機能を呼び出し、ハンドラを登録して受信メッセージを処理します。

=== WASM 対応

Colonio の Node は Go の WebAssembly ビルドをサポートしています。
WASM としてビルドした Node をブラウザ上の JavaScript アプリケーションに組み込むことで、ブラウザも P2P ネットワークに参加できます。
ブラウザ Node はブラウザネイティブの WebRTC API を使用して他の Node と接続するため、ネイティブ Node との互換性があります。

== ネットワーク層

Node のネットワーク層は以下のモジュールで構成されます。

SeedAccessor::
Seed との通信を担当します。
Connect RPC のストリーミングを使い、シグナリングメッセージの送受信と Keepalive を処理します。

NodeAccessor::
他の Node との WebRTC 接続を管理します。
接続の確立・維持・切断、およびパケットの送受信を行います。

Transferer::
パケットの送受信を統括するモジュールです。
リクエスト/レスポンスのマッチング、リトライ制御、パケットタイプごとのハンドラへのディスパッチを行います。

Routing::
パケットの宛先に応じた次ホップの決定を行います。
1D ルーティングと 2D ルーティングの2種類があります。

=== 1D ルーティング

NodeID をリング状の空間として扱い、宛先 NodeID に最も近い Node にパケットを転送します。

各 Node は隣接ノードの情報を保持し、段階的に離れたノードへの経路も管理します。
NodeID 空間を対数的に分割したレベル(8段階)で経路を管理することで、
ノード数 N に対して少数のリンクで効率的なルーティングを実現しています。

1D ルーティングは Messaging やパケット転送の基盤として使用されます。

=== 2D ルーティング

各 Node が 2D 座標上の位置を持ち、座標ベースで最も近い Node にパケットを転送します。

隣接 Node から受信した位置情報に基づいてルーティングテーブルを構築します。
隣接関係の計算には Delaunay 三角分割を使用し、平面または球面上でのボロノイ近傍を求めます。

2D ルーティングは Spread 機能の基盤として使用されます。

=== 座標系

2D ルーティングでは以下の2つの座標系を選択できます。

Plane(平面)::
矩形の平面空間です。X/Y の最小値・最大値で空間の範囲を定義します。

Sphere(球面)::
球面空間です。半径で空間の大きさを定義します。
緯度・経度を使った地理的なアプリケーションに適しています。

=== パケット

パケットは Protocol Buffers で定義されたフォーマットで送受信されます。
各パケットには送信元 NodeID、宛先 NodeID、ホップカウント、モードフラグ、および内容(Content)が含まれます。

ホップカウントは中継のたびにインクリメントされ、上限(デフォルト64)に達したパケットは破棄されます。
これによりルーティングループによるパケットの無限循環を防止します。

=== トポロジー維持と自己修復

P2P ネットワークでは Node の参加・離脱が常に発生します。
Colonio では Seed と Node が協調してトポロジーを維持・修復します。

==== Node 参加時

新しい Node がネットワークに参加すると、Seed から既存 Node を紹介されて WebRTC 接続を確立します。
接続が確立されると、隣接 Node 間でルーティング情報を交換し、1D・2D ルーティングテーブルを更新します。
隣接 Node の変化は接続先の再計算をトリガーし、必要に応じて新たな接続が追加されます。

==== Node 離脱時

Node が正常に終了する場合は Seed に切断を通知し、Seed がネットワークから除去します。
Node が異常終了した場合は Keepalive が途絶えるため、Seed が有効期限切れで離脱を検知します。

Node 側でも WebRTC 接続の切断を検知すると、ルーティングテーブルから該当 Node を除去し、接続先を再計算します。
さらに、切断を検知した Node は Seed に対して切断された Node の生存確認(Keepalive リクエスト)を要求します。
これにより、Seed 側の有効期限切れを待たずに早期の離脱検知が可能になります。

==== Reconcile

各 Node は自身の隣接ノードリストを Seed に定期的に報告します(ReconcileNextNodes)。
Seed は報告されたリストと実際に登録されているノードを照合し、一致していれば安定状態(Stable)を返します。
安定状態になると、Node はルーティングが正しく収束したことを確認できます。

== 機能: Messaging

Messaging は NodeID を指定した 1 対 1 のメッセージ送信機能です。

=== 動作

. 送信側 Node が宛先 NodeID とメッセージ名、ペイロードを指定して送信する
. パケットは 1D ルーティングにより宛先 Node まで転送される
. 宛先 Node のハンドラが呼び出され、レスポンスを返す
. レスポンスが送信元 Node に返される

=== オプション

AcceptNearby::
宛先 NodeID に完全一致する Node が存在しない場合、
NodeID 空間上で最も近い Node がメッセージを受信します。
これにより、特定 NodeID を担当するノードへのリクエストが可能になります。

IgnoreResponse::
レスポンスを待たずに送信を完了します。

== 機能: Spread

Spread は 2D 座標上の指定範囲内にある全 Node にメッセージをブロードキャストする機能(ジオキャスト)です。

=== 動作

. 送信側 Node が中心座標、半径、メッセージ名、ペイロードを指定して送信する
. 自ノードが範囲内にいる場合、隣接 Node のうち範囲内にある Node にリレーする
. 自ノードが範囲外にいる場合、2D ルーティングで中心座標に最も近い Node にリレーする
. 各 Node は受信済みメッセージのキャッシュを保持し、同一メッセージの重複処理を防止する

=== 最適化

Knock パケット::
大きなペイロードを送信する前に、相手 Node にキャッシュの有無を確認する小さなパケットを送ります。
既にキャッシュが存在する場合は本体の送信を省略し、帯域を節約します。
`SpreadSizeToUseKnock` で閾値を設定できます(デフォルト: 4096バイト)。

=== オプション

SomeoneMustExists::
範囲内に少なくとも1つの Node が受信したことを保証します。
範囲内に Node が存在しない場合はエラーを返します。

== 設計上の制約

=== Seed の単一障害点

デフォルト構成では Seed は単一プロセスで動作し、単一障害点(SPOF)となります。
Seed がダウンした場合、新規 Node の参加や WebRTC シグナリングができなくなります。
ただし、既に確立済みの Node 間の P2P 通信は Seed を経由しないため、影響を受けません。

Seed を冗長化するには、SessionStore と Gateway の両方を分散対応の実装に差し替える必要があります。
Gateway インタフェースは Publish/Subscribe パターンで設計されているため、DB やメッセージブローカーを使用した実装に差し替えることで、複数の Seed インスタンス間でシグナリングメッセージやノード管理を共有できます。

=== ルーティングの収束時間

Node がネットワークに参加または離脱した直後は、ルーティングテーブルが最新の状態に収束するまで時間がかかります。
ルーティング情報は隣接 Node 間で段階的に伝播するため、ネットワーク全体に反映されるにはノード数に応じた遅延が発生します。

収束が完了するまでの間、一部のパケットが最適でない経路を通ったり、宛先に到達できない可能性があります。
各 Node は Seed との Reconcile により自身のルーティングが安定状態(Stable)かどうかを確認できます。
安定状態に達するまでは、KVS のように整合性が求められる操作は抑制されます。
